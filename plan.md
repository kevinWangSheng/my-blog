## 1. 主目标

最终目标不是传统技术博客,而是一个**个人知识空间 / 公开自留地 + 简历/作品展示入口**。

它的核心目标是:让 Shawn 能长期、低摩擦地公开沉淀自己的学习、思考、项目和阶段性表达;同时作为类似简历/作品集的对外入口,让陌生读者可以快速理解「这个人是谁、在关注什么、做过什么、具备什么能力、正在形成哪些判断」。

换句话说,它不是先服务流量、SEO、教程体系或旧站迁移;它先服务一个长期可生长的个人公开空间。

## 2. 最终形态应该支持的内容形态

最终形态至少要能容纳这些内容;实现可以分阶段,但目标设计不按临时 MVP 降级。

### 2.1 关于我 / 身份入口

- 说明 Shawn 是谁、当前关注什么、正在学习/构建什么。
- 给陌生读者一个 30 秒内可理解的入口。
- 不做成僵硬简历,但可以承载作品、方向和联系方式。

### 2.2 学习笔记 / 知识卡片

- 支持不完整、持续更新的笔记。
- 允许短内容、半成品、概念澄清、学习路径记录。
- 不强迫每条内容成为完整文章。

### 2.3 项目 / 构建记录

- 记录做过或正在做的项目。
- 重点不是展示最终成品,而是记录目标、过程、决策、踩坑和结果。
- 适合承载 AI agent 工作流、学习系统、工具项目等长期线索。

### 2.4 阶段性输出

- 支持周报 / 月报 / 阶段复盘 / 近期关注。
- 用来把散落的学习和项目进展定期收束。
- 可以成为读者理解近期状态的入口。

### 2.5 成熟长文

- 当某个主题足够成熟时,再沉淀成长文。
- 长文是结果,不是默认生产单位。
- 不以教程站方式追求覆盖面,更重视个人判断和经验密度。

### 2.6 连接层

- 支持 RSS。
- 支持友链 / 推荐阅读 / 外部链接集合。
- 这部分用于接入中文独立站生态;可以轻量起步,但需要作为长期能力保留。

## 3. 最终目标不追求的东西

- 不做传统技术教程站。
- 不以 SEO 流量为主要目标。
- 不以商业变现、订阅会员、广告为目标。
- 不追求复杂评论系统、登录系统、社区功能。
- 不迁移旧站全部内容;旧内容默认可废弃,有价值的单篇只作为素材重写。
- 不先做复杂的图谱、双链、可视化关系网络;如果需要,后续再加。
- 不为了炫技选择复杂技术栈。

## 4. 读者与使用场景

最终形态优先服务两个读者:

1. **自己**:作为长期学习、项目推进、思考沉淀的公开界面。
2. **有缘读者 / 外部评价者**:包括朋友、未来合作者、招聘/合作方、同方向学习者。他们不一定从搜索进来,更可能从链接、RSS、友链、GitHub、简历或聊天中进入。

读者进入后应该能完成三件事:

- 快速知道 Shawn 当前的方向。
- 找到正在生长的学习/项目线索。
- 看到一些已经沉淀过的成熟判断或作品。

## 5. 信息架构的目标,不是具体排版

这一阶段只定目标,不定 UI 排版。后续设计时,信息架构至少要满足:

- 首页同时承载「我是谁 / 能力 / 作品」与「最近在学什么 / 做什么 / 正在形成什么判断」。
- 有一个清晰的身份入口,但不把全站压成纯简历。
- 有一个「正在生长」的区域,承载笔记、卡片、项目过程。
- 有一个「已经沉淀」的区域,承载长文、复盘、作品。
- 有一个连接层,承载 RSS、友链、外部链接或推荐。
- 内容可以逐步生长,而不是上线当天就必须完整。

## 9. Astro 技术蓝图草案

> 目标:把已定方向翻译成可实施骨架;本节仍属于 ③规划,不是实现任务。

### 9.1 新站目录原则

新站不在旧 Hugo 文件上原地改造。首个实现任务应创建独立 Astro 新站目录,再逐步切换。建议目录意图如下:

```text
/
  AGENTS.md
  DECISIONS.md
  plan.md
  tasks.md                 # 到 ④任务拆分阶段创建
  site/                    # Astro 新站根目录,首个实现任务创建
    src/
      content/             # Astro content collections
        notes/
        logs/
        essays/
        projects/
        links/
      pages/               # 路由页面：首页、about、各集合列表/详情、rss.xml
      layouts/             # 页面壳、内容页布局
      components/          # UI 组件,不绑定整站主题
      styles/              # 全局样式与设计 token
      lib/                 # 内容查询、排序、slug、日期等辅助函数
    scripts/               # agent 可调用的同步/检查工具
    public/                # 静态资源
```

说明: `site/` 作为并行新站根目录,是为了避免误动旧站文件和旧部署链路。

### 9.2 Content collections 草案

Astro 侧使用 content collections / Content Layer API 管理内容。一级内容类型与集合映射:

| 内容类型 | 集合/页面 | 用途 |
|---|---|---|
| About | `src/pages/about.astro` 或 `src/content/about/` | 身份、方向、能力、联系方式 |
| Projects | `src/content/projects/` | 作品、项目、构建记录 |
| Notes | `src/content/notes/` | 知识卡片、学习笔记、半成品想法 |
| Logs/复盘 | `src/content/logs/` | 周/月报、阶段记录、过程复盘 |
| Essays | `src/content/essays/` | 成熟长文、判断、总结 |
| Links | `src/content/links/` 或 data 文件 | 友链、推荐阅读、RSS/独立站连接层 |

暂不引入复杂成熟度状态;如果后续需要,再增加轻量字段。

### 9.3 Front matter 最小字段

所有可发布内容建议共享一组最小字段:

```yaml
title: string
description: string
date: YYYY-MM-DD
updated: YYYY-MM-DD # 可选
tags: [string]
source: string # 可选,来源说明或外部源链接
visibility: public # 预留,默认 public
```

不同集合追加少量字段:

- `projects`: `status`、`role`、`repo`、`demo`、`period`。
- `notes`: `topic`、`related`。
- `logs`: `period`、`summary`。
- `essays`: `canonical`、`series`。
- `links`: `url`、`category`、`feed`、`note`。

字段原则:少而稳定,优先让 agent 容易生成,不要让发布工具为了填字段而阻塞。

### 9.4 Agent output 中间格式

由于内容来源不限制,blog 项目应接收一个稳定的 agent output。建议初始支持两种输入:

1. 单篇 Markdown 文件:带 YAML front matter +正文。
2. 一个 output 目录:包含若干 `.md` 文件和一个可选 `manifest.json`。

建议 `manifest.json` 形态:

```json
{
  "items": [
    {
      "type": "notes|logs|essays|projects|links",
      "file": "relative/path/to/file.md",
      "slug": "optional-slug",
      "publish": true
    }
  ]
}
```

同步工具根据 `type` 写入对应 content collection。

### 9.5 同步工具草案

项目内提供 agent 可调用命令,不绑定 Claude Code 或 Codex。命令形态后续可调整,初始目标如下:

```bash
# 检查单篇或 output 目录,不写入
pnpm content:check <path>

# 同步单篇或 output 目录到 Astro content collections
pnpm content:sync <path>
```

`content:check` 至少检查:

- 是否包含疑似 secrets / token / key。
- 是否包含明显私密标记或禁止发布标记。
- front matter 是否满足最小字段。
- `type` 是否能映射到已知集合。
- slug/path 是否安全、是否冲突。
- 内容是否基本可读:标题、摘要、正文非空。

`content:sync` 在 check 通过后执行:

- 规范化 slug 和文件名。
- 补齐缺省字段。
- 写入 `site/src/content/<collection>/...`。
- 输出同步报告,供 agent 继续提交或修正。

### 9.6 RSS 与生态连接

Astro 新站需要提供 RSS。初始策略:

- 先生成全站综合 feed。
- feed 来源优先包含 `essays`、`logs`、必要时包含公开 `notes`。
- `projects` 是否进入 RSS 后续按内容密度决定。
- `links` 页面保留友链 / 推荐阅读 / 外部 RSS 入口。

### 9.7 部署与 cutover 原则

- 旧站内容和部署链路保持冻结;cutover 前不改旧 Hugo CI / 部署分支。

## 11. 首页信息架构推荐方案

> 来源:按用户要求由 subagent 给出建议,主 agent 采纳为默认规划。此处只定结构顺序,不定 UI 样式。

### 11.1 默认首页区块顺序

1. **Hero / 身份入口**
   一句话说明 Shawn 是谁、正在研究什么、这个站有什么价值;提供 About / Projects / Notes 等主要入口。

2. **当前关注 / Learning Now**
   展示最近正在学习、研究、整理的主题,链接到 Notes / Logs / Essays。

3. **Featured Projects**
   精选 2–4 个作品或项目,作为能力与简历展示入口。

4. **Recent Notes / Logs**
   展示少量最近同步/整理出的学习笔记、过程记录、复盘,但不做传统博客时间线。

5. **Essays**
   展示成熟长文或阶段性判断,作为深度资产入口。

6. **Links / Reading Garden**
   展示关注的信息源、工具、朋友、参考材料、友链/RSS 生态连接。

7. **Footer / Contact**
   提供 GitHub、邮箱、RSS、简历、社交链接等稳定出口。

### 11.2 排序理由

先建立身份,再证明这个人正在学习和创造,最后提供深入探索路径。这样避免首页变成传统文章列表,也避免变成冷冰冰的 portfolio。

### 11.3 备选方案

- 更偏公开学习空间:把 `Learning Now` 权重提高,紧贴 Hero 并弱化 Projects。
- 更偏招聘/合作:把 `Featured Projects` 提到第二位,`Learning Now` 第三位。

### 11.4 风险

- 自动同步内容太多时,首页可能低信噪比。首页只展示精选和最近少量内容,完整流留给列表页。
- Projects 长期不维护会削弱可信度。
- Notes / Logs / Essays 边界不清时,访客会不知道该点哪里;实现时需要用文案解释三者差异。

## 12. UI / 审美方向草案（历史记录,已被返工任务覆盖）

> 历史说明:本节记录首轮“技术炫酷 / AI Lab Manual”方向的探索,不再作为当前实现验收基线。当前方向以 `DECISIONS.md` 的 `D-20260614-UI审美方向修正`、`D-20260614-UI审美方向取代关系` 和 `tasks.md` T13–T17 为准:先保证 blog 基本功能与阅读链路,再补内容密度,再做清爽、宁静、留白充足、阅读优先的视觉重构。

### 12.1 核心判断

当前第一版 UI 不应只是“能读的博客模板”,也不应只是“能展示项目的简历页”。它应该是一个安静、可信、长期可阅读的个人知识空间:信息清楚、阅读舒服、内容入口明确;可以保留少量 agent/lab 结构感,但不能为了技术气质牺牲 Markdown 阅读和长期维护。

基础目标:

- 信息一眼能读懂:访客知道 Shawn 是谁、这里有什么、从哪里开始。
- 内容层级清楚:学习中、项目、近期记录、成熟长文、连接层各有位置。
- 布局合理:首页不是时间线,也不是卡片堆砌;每个区块都服务一个叙事节点。
- 审美优先:视觉完成度优先于功能堆叠,但不牺牲可读性、性能和维护成本。
- 安静的技术感:视觉语言可以传达“正在研究 AI / agent / 系统 / 工程构建”,但必须服务阅读和内容层级。

### 12.2 可选视觉路线

以下三条路线是历史探索记录。当前主方向已由 ⑦ human 反馈修正为清爽、宁静、留白充足、阅读优先的现代知识花园 / 安静实验室;B 不再是默认推荐。

#### A. Cold System / 冷峻系统界面

像一个被精心设计过的个人控制台:深色背景、细网格、冷色高光、数据面板感、精确排版。适合强调工程能力、系统思维和作品可信度。

- 视觉关键词:深色、冷光、细线、网格、坐标、状态指示、面板层级。
- 优点:技术感强,Projects 和构建记录会很有力量。
- 风险:容易变成 dashboard 或后台系统,个人感和阅读温度不足。
- 适合条件:如果第一版最想让人感到“这个人很会构建复杂系统”。

#### B. AI Lab Manual / AI 实验手册（历史探索,不再默认）

像一个未来实验室里的公开研究手稿:有技术光感和界面层次,但仍以文字、笔记和项目为核心。它不是纯后台,也不是纯文章站,而是“研究中的人 + 正在生成的系统”的混合空间。

- 视觉关键词:暗色或深灰基底、荧光点缀、实验卡片、扫描线/噪声纹理、局部发光、精致 Markdown 排版。
- 优点:同时容纳 Notes、Logs、Essays 和 Projects;既酷,又不丢个人学习空间的生长感。
- 风险:如果光效和卡片太多,会滑向常见 AI landing page;实现时必须克制紫蓝渐变和玻璃拟态。
- 适合条件:如果第一版想让人感到“这个人持续研究 AI 和工具系统,而且审美在线”。

#### C. Future Archive / 未来档案馆

像一个长期积累的私人技术档案馆:更偏内容沉淀和时间纵深,用索引、编号、档案标签、章节感和精致排版制造高级感。技术炫酷更内敛,不靠强动效。

- 视觉关键词:深浅混合、编号系统、档案标签、索引栏、长文阅读、稀疏高亮。
- 优点:非常适合长期知识库、复盘、长文和链接收藏。
- 风险:首屏冲击力弱于 A/B;如果处理不好会像文档站或知识库模板。
- 适合条件:如果第一版最想让人感到“这里会长期积累高质量判断”。

### 12.3 默认视觉气质

当前方向: **清爽、宁静、留白充足、阅读优先的现代知识花园 / 安静实验室**。旧 AI Lab Manual / AI 实验手册方向已被 ⑦ human 反馈修正,仅作为历史探索参考。

它更接近一个安静、可信、长期可阅读的个人知识空间,而不是强冲击实验舱、传统技术博客、Notion 公开页、普通作品集或 SaaS landing page。首页应该先保证阅读入口、内容层级和项目入口清晰;可以保留少量 agent/lab 结构感,但不能让视觉强度压过文字。

关键词:

- 技术感:网格、坐标、状态、编号、微型元信息、命令/输出隐喻。
- 宁静:低饱和强调色、舒展留白、轻柔动效、稳定阅读节奏,避免强荧光和过重背景。
- 精致:字体、行距、边框、分隔线、日期、标签、链接 hover 等细节要有设计感。
- 个人感:通过栏目命名、短句、精选内容、阅读路径体现,不靠大段自我介绍。
- 作品感:Projects 区块要有可信的展示强度,像把项目放进实验台中央展示。

### 12.4 2D / 3D 方向判断

当前采用 **2D 阅读界面为主 + 极轻量结构装饰为辅**。

原因:这个站的主体是 Markdown 内容、项目说明和长期阅读;当前 human 更偏好清爽宁静,因此不追求 3D 冲击或技术炫酷。第一版用 2D 承载信息结构,只允许非常轻的空间层次、分隔线、卡片阴影或局部装饰辅助阅读。

可用形式:

- 轻量结构装饰:细分隔线、柔和卡片阴影、低饱和状态标签、留白层级。
- 轻量 SVG / CSS:只用于辅助结构和动效,不做粒子/扫描线/强 HUD。
- 不使用真 3D 作为第一版必要能力;若未来要引入重动画或 3D,需另走新增依赖确认。

验收标准:第一眼清爽、宁静、可信,正文页像优秀阅读界面;移动端关闭或弱化装饰后不能损失信息表达。

### 12.5 视觉上避免什么

- 避免传统技术博客模板:大标题 + 分类标签 + 文章时间线不是首页主结构。
- 避免纯知识库模板:不要像公开 Obsidian/Quartz 一样只有节点、目录和反链感。
- 避免简历站模板:不要首屏大头像、技能条、Logo 墙、夸张 CTA。
- 避免通用 AI 审美:紫蓝渐变、玻璃卡片、满屏圆角卡、无语义图标堆叠。
- 避免廉价赛博朋克:不要 neon 过载、终端雨、黑绿黑客风、无意义 HUD 装饰。
- 避免只炫不读:Notes、Logs、Essays 都是 Markdown 内容,正文阅读体验必须成立。

### 12.6 首页体验方向

首页第一眼应该给人的感受是:这是一个清爽宁静、长期积累的个人 AI / 技术知识空间;这个人正在认真学习、构建和形成判断,而且这个空间适合阅读。

推荐节奏:

1. **Hero 像安静入口,不是广告横幅**
   Hero 用一句有信息量的自我定位 + 一小段站点说明。视觉可以有细分隔、低饱和状态信息或“current focus”提示,但不要做空洞大口号或压迫性大标题。

2. **Learning Now 像实时研究面板**
   展示 3–5 个当前主题,每个主题用短句说明“为什么在看”。可以使用状态、编号、topic chip、progress-like 视觉,但不要做成普通标签云。

3. **Projects 像实验台上的重点装置**
   只放精选项目。每个项目强调“问题 / 做了什么 / 结果或状态”,视觉上给更强版面权重,让外部读者能快速判断能力。

4. **Notes / Logs 像运行日志和田野记录**
   展示少量最近内容,弱化传统时间线感,强调“正在生成”。它不抢首页主叙事,但让站点显得活着。

5. **Essays 像沉淀报告**
   数量可以少,但版面要稳。成熟长文不需要多,需要被看见;视觉上应从“运行中”切到“已沉淀”。

6. **Links 像外部信号源**
   这里可以更有个人品味:工具、信息源、朋友、推荐阅读、RSS。它是审美和价值观的侧影,不是友情链接墓地。

### 12.7 页面类型体验方向

- `About`:不是完整简历,而是“如何理解我”的入口。可以包含方向、能力、正在关注、联系方式、代表项目链接;视觉上像个人系统说明页。
- `Projects`:要比普通列表更强展示。第一版可用精选卡片 + 状态 + 简短 case note,后续再加详情页深度;视觉上最应该有技术作品感。
- `Notes`:轻量、可浏览、低压力。视觉上允许短内容成立,像实验记录或知识片段,不要逼每条笔记像正式文章。
- `Logs`:强调时间和过程,但避免传统 blog archive 味道。适合阶段复盘、周/月报、项目过程;可以有 run log / changelog 的隐喻。
- `Essays`:阅读体验最重要。宽度、行距、标题层级、引用、代码块、脚注/链接都要认真处理;技术感只能做边框和元信息,不能干扰正文。
- `Links`:更像 curated signal page。按主题分组,每个链接最好有一句个人理由。

### 12.8 第一版设计原则

- 先做一个有审美完整度的轻量系统,不要一开始做复杂主题系统。
- 字体与排版仍是第一设计对象;技术感来自排版、层级、颜色、边框和微交互,不是堆特效。
- 优先采用清爽宁静、阅读友好的浅色或柔和深浅混合主题;不以深色荧光作为默认。
- 色彩要有明确主调:柔和底色 + 低饱和强调色 + 稳定正文色,不要彩虹式标签系统或强荧光过载。
- 组件数量少而精:Hero、Section、ContentCard、ProjectCard、NoteItem、EssayItem、LinkGroup、Footer。
- 动效服务“阅读节奏 / 内容层级 / hover 反馈”;允许使用克制 scroll reveal:下滑时区块柔和渐显、轻微位移或层级展开,上滑返回时可自然淡出或回到待激活状态。
- scroll reveal 必须克制且可读:不能让读者为了看内容被迫等待动画,不能每个元素都乱飞;核心文字应快速可见,装饰层可以更有仪式感。
- 第一版优先使用 CSS/IntersectionObserver 实现轻量滚动动效;只有在视觉明显受益时才引入更重的动画库。
- Markdown 内容必须自然成立:短笔记、长文、项目说明都不能因为缺图而显得破。

### 12.9 用于实现阶段的验收标准

实现后自验证不仅要跑 build,还要截图检查这些点:

- 首页首屏是否清爽、宁静、阅读优先,不是默认模板感。
- 不看导航时,访客是否能在 5 秒内理解这个站的用途。
- 首页是否同时呈现“学习自留地”和“作品/能力入口”,且不互相打架。
- 安静的技术感是否来自有效排版、内容层级和细节,而不是廉价特效或模板化 AI landing page。
- Markdown 长文是否舒服可读。
- 短 note 是否不会显得太空或太像 bug。
- Projects 是否有足够展示强度。
- 移动端是否仍然有排版美感,不是简单纵向堆叠。

### 12.10 内容样本策略

第一版实现可以先用少量手写示例内容验证 UI,不必先接入真实知识库。需要更接近真实效果时,再从个人知识库或近期 output 中抽取:

- 1 个项目样本。
- 2–3 条 Notes。
- 1 条 Log / 复盘。
- 1 篇 Essay 或长文占位。
- 3–5 个 Links。

抽样目标不是填满站点,而是测试 UI 能否承载真实 Markdown 内容的长度、密度和不规则性。
