Agent memory 不是向量库，是写入策略

真正有用的 agent memory 不只是保存聊天历史,而是决定什么状态值得写入、更新、选择、遗忘和审计。

如果把 agent memory 理解成“给 agent 接一个向量数据库”,这个系统从一开始就会偏。

向量检索能帮 agent 找到相似材料,但 memory 真正困难的部分不是“存在哪里”,而是:什么状态值得写入? 谁有权写? 旧事实如何失效? 冲突如何保留 trace? 下一轮任务只应该选哪些 memory 进上下文? 人如何看见、删除、回滚和审计这些记忆?

所以我更愿意把 agent memory 定义成一种系统层能力:agent 在长任务、跨会话、跨工具、跨人类协作中保存、更新、选择、压缩、检索和治理状态的能力。

Memory 不是 chat history

Chat history 是事件日志。它记录发生过什么,但不自动变成可靠知识。

一个对话里可能有临时偏好、过期判断、错误尝试、被推翻的方案、一次性的调试输出。直接把整段历史塞进下一次上下文,会让 agent 看起来“记得很多”,但实际上是在把噪声延长寿命。

更好的 memory 系统至少要区分四类东西。

第一类是当前任务状态:目标、约束、计划、失败尝试、验证结果、下一步。这类 memory 生命周期短,但准确性要求很高。它回答的是“任务能不能继续”。

第二类是长期事实:用户偏好、项目约定、实体关系、时间变化。这类 memory 需要来源、更新时间、冲突处理和失效机制。它回答的是“这个事实现在还成立吗”。

第三类是 episodic trace:上次发生了什么、怎么解决、哪里失败。它不一定应该直接进入下一轮 prompt,但很适合在相似任务出现时被召回。

第四类是 procedural memory:以后应该怎么做。项目里的 AGENTS.md、技能文件、检查清单、发布流程、反模式记录,都属于这种形态。它比“我记得你喜欢 X”更像真正的经验积累。

判据是 write path

我现在判断 RAG 和 memory 的最短判据是 write path。

RAG 通常是只读的外部证据供应链。它从文档、网页、知识库、数据库里找当前问题所需的材料,再装配进上下文。它可以很复杂,但核心仍然偏 read path。

Memory 一定有 write path。系统会从自己的运行、互动、失败和人类反馈中写回状态。它不只读取世界,还改变自己未来会看到什么。

这也是为什么“数据量”不是判据。一个 50k 文档库,如果每晚从外部源重建,agent 从不写它,那仍然更像 RAG。一个只有十条偏好的小文件,如果 agent 会根据长期互动更新它,那就是 memory。

Memory 的难点不是召回相似文本,而是治理系统如何改变自己。

写入比检索更危险

Memory write policy 是最容易被低估的部分。

如果写入太激进,agent 会把临时偏好、一次性约束、误解甚至 prompt injection 写成长期状态。下一次任务再读出来,错误就获得了持久性。

如果写入太保守,agent 又永远不会积累经验。每次都像第一次合作,人必须反复解释同样的边界和习惯。

比较健康的策略是让 memory 写入有层级:短期任务状态可以自动保存;长期偏好需要来源和时间;会影响外部行为的规则应该可见、可审计、可回滚;涉及隐私、权限或组织策略的 memory 应该有更强的人类治理。

Selection 才决定它有没有用

存下 memory 不等于 agent 会用好 memory。

每一轮上下文都有限。真正的工程问题是 selection:当前任务需要哪些 memory,哪些只是历史噪声,哪些应该被压缩,哪些应该被隔离。一个 memory store 如果只能“全部塞进去”或“按相似度取 top-k”,很快会遇到和 naive RAG 类似的问题。

这也是为什么 2026 的 agent memory 讨论越来越像 context engineering。OpenAI Dreaming、LangGraph memory taxonomy、Anthropic managed agents 这类信号都在指向同一件事:memory 不只是一个后端存储模块,而是运行时上下文管理的一部分。

好 memory 应该让 agent 像有经验的同事

我对 agent memory 的期待不是“它记得更多细节”,而是“它少犯同样的错”。

有经验的同事不会把所有会议记录背下来。他会知道哪些决策已经定了,哪些偏好只是临时的,哪些错误下次要避开,哪些流程必须遵守,哪些地方需要重新确认。

所以 memory eval 也不应该只测 recall。更有意义的问题是:

  • agent 是否能沿用上次已经确认的项目约定?
  • agent 是否会把过期事实当成当前事实?
  • agent 是否能从失败经历中形成 procedural rule?
  • agent 是否能在不暴露私密信息的情况下复用经验?
  • human 是否能看见、修正和删除错误 memory?

如果这些问题没有答案,那所谓 memory 很可能只是一个更长、更隐蔽的 chat history。

真正的 agent memory 不是“记住一切”。它是让系统知道什么值得留下,什么应该忘掉,以及什么时候必须回到人那里确认。
