公开知识花园最难的地方,不是生成页面,而是让内容长期可生长。一个人每天都会产生很多碎片:读到的文章、项目里的选择、调试时发现的坑、对某个概念的临时判断。如果每个碎片都要求被写成完整文章,发布会变成负担;如果所有碎片都原样公开,读者又会被噪声淹没。

## Agent 适合做整理层

Agent 的价值在于把“整理到可发布”这段路变短。它可以从项目日志里抽出一条 weekly log,从学习笔记里整理出 note,从一组已验证材料里起草 essay,也可以在发布前检查 frontmatter、slug、RSS、疑似 secrets 和发布禁用标记。它做的是低摩擦的结构化工作,不是替人决定什么值得公开。

这很像一个编辑助理。它可以提醒你这段文字缺少来源,这条 note 太短,这个项目页没有说明 role 和当前状态,这篇文章可能混入了未确认判断。但最后决定内容是否进入公开站点的人,仍然应该是 human。公开边界不是技术细节,而是责任边界。

### 内容类型本身就是读者体验

这个站点把内容分成 Essays、Notes、Logs、Projects 和 Links,不是为了复杂,而是为了降低读者理解成本。

- Essays 是相对完整的长文综合;
- Notes 是短判断、知识卡片或正在形成的概念;
- Logs 是阶段过程,保留项目和学习如何变化;
- Projects 是能力证据入口,需要说明问题、角色、方案和进展;
- Links 是来源货架,解释为什么某个公开资料值得复访。

当类型清楚时,读者不会误把半成品 note 当成最终文章,也不会把阶段 log 当成对外成果包装。站点的结构在替内容说明成熟度,这比每篇文章开头反复解释“这只是草稿”更自然。

> 好的个人站不是把所有东西发布出去,而是让公开内容处在合适的位置上。

## 为什么不能只剩自动化

如果发布流完全自动化,风险会被放大。Agent 可能把私人项目路径、未确认的判断、旧材料里的过期信息,甚至不适合公开的上下文整理成看似漂亮的页面。越是流畅,越容易让人忽略它是否应该公开。

所以第一版站点更应该先把安全地板做扎实:显式 manifest、content check、sync 前失败不写入、RSS 可验证、构建可复现、本地 UI 验证不过不进入 human 验收。自动化可以扩大产能,但它必须被检查约束。没有约束的自动发布,只是更快地产生风险。

## 一个更理想的回路

理想状态下,知识库、项目日志和博客不是三套割裂系统。项目推进时留下 log;学习材料整理成 note;多个 note 稳定后沉淀成 essay;项目里被证明有价值的能力再进入 Projects。Agent 在这些节点之间搬运和整理,人只在关键判断点介入:这是否准确,是否公开,是否代表当前方向。

这样的公开知识花园不会每天都产出完美文章,但它会持续留下可读的生长痕迹。对陌生读者来说,这比一个静态简历更真实;对自己来说,它也比一堆永远不会回看的私人碎片更有复利。
