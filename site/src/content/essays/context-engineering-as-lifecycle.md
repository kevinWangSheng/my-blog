很多关于 context engineering 的讨论,会很自然地从窗口长度开始。窗口越长,能塞进去的文档越多,模型看起来也越不容易“忘”。但如果把上下文只理解成一个更大的容器,系统很快会遇到另一类问题:噪声也被保存得更久,过期结论也更容易混进当前判断,而真正应该保留的状态反而没有明确位置。

## 上下文不是仓库,而是运行时

一个 agent 在执行任务时,上下文更像运行时内存,不是永久知识库。它需要快速读取当前目标、约束、证据、最近失败和下一步行动,但它不应该无差别保存所有历史。否则上下文会变成一个热闹但没有索引的房间:什么都在,却很难知道什么仍然重要。

更稳定的做法是把上下文拆成生命周期。信息进入系统时,先判断它是短期工作记忆、长期项目状态、可复用知识,还是一次性证据。短期工作记忆可以在任务结束后丢弃;长期项目状态应该写回 repo 文件或任务板;可复用知识应该进入知识库;一次性证据则只需要在验收摘要里留下路径和结论。

### 五个动作

我更愿意把 context engineering 看成五个动作的组合:

- 写入:什么东西值得进入上下文,以什么结构进入;
- 选择:当前任务只需要哪些事实、约束和历史决策;
- 压缩:长材料如何保留判断所需的骨架,而不是只做摘要;
- 隔离:哪些内容不应该混进当前执行,例如旧方向、私密材料或未确认推断;
- 恢复:新 agent 接手时,如何从稳定文件里重新获得足够状态。

这里的关键不是“压缩得越短越好”,而是让压缩后的内容仍然能支持行动。比如一句“之前验证过 UI”没有意义;更好的状态是:验证命令是什么、summary 路径在哪里、失败过什么、现在还有什么未确认。

> Context 的质量不取决于它有多满,而取决于它是否让下一步行动更准确。

## 在个人站里的体现

这个博客重建也在实践同样的原则。`AGENTS.md` 只放长期稳定规则,`DECISIONS.md` 保存已定决策,`tasks.md` 保存当前执行队列,`out/acceptance/` 保存本地验收证据。它们不是文档洁癖,而是在给未来的 agent 留可恢复状态。

如果所有计划都散在对话里,下一个 agent 就只能靠猜。如果所有内容都塞进一个巨大说明文档,真正当前的任务又会被历史噪声淹没。分层的好处是,每一种信息都有自己的寿命:长期规则慢慢变,任务状态快速变,验收证据只服务当前闭环,完成后可以归档。

## 风险:把压缩当成理解

压缩还有一个常见陷阱:摘要看起来流畅,但它可能丢掉了冲突、例外和证据粒度。对于学习系统来说,这尤其危险。一个 agent 可以把三篇文章合并成一段漂亮总结,却把它们真正不同的前提抹平。这样的压缩会让人更快读完,也更快误判。

所以我希望系统保留的不只是结论,还有结论的边界。哪些是 checked facts,哪些是 inference,哪些只是 recommendation;哪些内容来自官方文档,哪些来自经验文章;哪些判断会随着时间漂移,需要重新验证。这些边界让上下文不只是信息,而是可校准的状态。

当上下文被这样设计,agent 才不只是“记住更多”,而是能在长任务里更稳地接力。它知道哪些东西该带走,哪些该放下,也知道什么时候应该回到人那里完成真正的判断。
