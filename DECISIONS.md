# DECISIONS.md

博客重建的决策日志(ADR-lite,**只追加**)。记录会影响实现路径、内容流程、部署、信息架构、站点风格的选择;小的文案 / 样式微调不记。`AGENTS.md` 只引用结论,全文在此。

格式:

```
### D-YYYYMMDD-短标题
- 状态:提议 / 已定 / 废弃
- 背景:为什么现在做这个选择
- 决策:选了什么
- 取舍:牺牲什么、得到什么
- 推翻条件:什么情况重新讨论
```

### D-20260613-规范选型
- 状态:已定
- 背景:博客体量小、solo、纯 agent 驱动,需要一套不重的规划规范。
- 决策:执行层用 `plan.md`(规划)+ `tasks.md`(任务,含验收标准);决策用本文件内联 `D-` 日志。**暂不上** OpenSpec / spec-kit / 独立 ADR 目录。
- 取舍:省掉工具与目录开销;代价是手动维护这几个 md。
- 推翻条件:上线后进入长期高频迭代,或单会话装不下一个变更的上下文 / 多变更并行互相干扰 → 把 `plan.md` 升级为 OpenSpec 的每变更隔离。

### D-20260613-技术栈选型
- 状态:已定
- 背景:新站目标是学习自留地 + 简历/作品展示入口,需要静态站、Markdown 友好、agent 易写入、多内容类型、RSS 易生成、UI 不被强主题锁死。
- 决策:使用 **Astro 自建静态站** 作为新站技术栈。后续以 Astro content collections / Markdown 内容为核心,不采用强绑定视觉形态的整站主题。同步工具把 agent 整理后的 output 转换为站点内容集合。
- 取舍:获得较高 UI 自由度、现代静态站体验、多内容类型建模能力和 agent 写入友好性;代价是相比套用 Hugo/Quartz 主题需要更多自定义实现。
- 推翻条件:如果后续发现 Astro 对内容同步、构建、部署或维护成本明显不合适,再评估 Hugo/Quartz/Next static export 等替代方案。

### D-20260613-旧站作为遗留
- 状态:已定
- 背景:重建前已有一个线上博客实现(当前 worktree 即其代码)。
- 决策:把旧站当**过去的实现**,不作为新站设计出发点;新站从目标倒推,不默认继承旧结构。重建期**冻结旧站**,不动其部署链路。
- 旧站非显然的坑(仅迁移 / 清理时参考,cutover 后失效,现在不要去「修」):

  - 远端有遗留分支 `add-claude-github-actions-…`,非活跃,勿误当主线。
- 推翻条件:决定保留旧站某部分(如内容源链路)时,把对应事实从「遗留」升格为 `AGENTS.md` 的正式约定。

### D-20260613-排除传统技术博客
- 状态:已定
- 背景:已有两轮外部调研和讨论。传统技术博客通常以时间倒序文章、分类/标签、教程/经验贴为中心,容易把项目导向「写完整技术文章」和 SEO/流量优化,不符合当前重建目标。旧站也不再作为新站设计起点。
- 决策:新站第一版**不做传统技术博客 / 教程站**。后续设计不以「文章列表 + 分类标签 + 技术教程」为默认结构。
- 取舍:牺牲传统博客的低理解成本、现成主题和搜索流量路径;换来更贴近个人长期学习、表达、整理和作品呈现的空间。
- 推翻条件:如果后续明确目标变成求职技术品牌、SEO 获客、系统化教程输出,再重新讨论是否引入传统博客区,但仍不作为全站唯一形态。

### D-20260613-第一版站点形态
- 状态:已定
- 背景:当前目标尚未细化到首页结构和技术栈,但已能确认新站不应围绕旧 Hugo/Notion 实现或传统技术博客模板展开。
- 决策:第一版方向定义为**个人知识空间 / 公开自留地**。它优先服务个人长期学习、思考沉淀、项目/作品呈现和阶段性表达;成熟内容可以成为长文,但长文不是唯一内容单位。
- 取舍:牺牲单一博客形态的简单性;换来可容纳短笔记、学习卡片、项目记录、长文、周报/月报、关于页和连接页的弹性。
- 推翻条件:如果后续发现主要使用场景只剩作品集 / 简历展示 / newsletter / 私人笔记,则把站点形态收窄到对应方向。

### D-20260613-后续调研采信标准
- 状态:已定
- 背景:前两轮 deepresearch 对市场形态有粗分类价值,但检索质量不足:英文轮混入平台/作品集/SEO 榜单,中文轮混入建站指南、CSDN 模板推荐、乱码知乎源和传统 IT 博客目录,不足以支撑首页结构、栏目命名、视觉风格、内容模型或技术栈决策。
- 决策:前两轮报告只作为**粗分类参考**,不作为最终设计证据。后续调研只采信真实个人站和高质量目录的站点级观察:首页组织、内容分区、更新节奏、RSS/友链/about/now/项目页、读者第一眼体验、内容单位和维护方式。
- 取舍:牺牲泛搜速度;换来更少 SEO 噪声和更贴近真实个人站生态的判断。
- 推翻条件:如果后续已有足够强的个人偏好,可以减少外部样本调研,直接进入规划原型。

### D-20260613-最终主轴
- 状态:已定
- 背景:最终目标需要同时服务长期学习沉淀与对外理解,但不能退回传统技术博客或纯简历站。
- 决策:新站最终主轴为**学习自留地 + 简历/作品展示入口**。学习自留地是内容主体,用于承载长期知识沉淀、项目过程、阶段复盘和个人判断;简历/作品展示是对外入口,帮助陌生读者快速理解 Shawn 的方向、能力和代表性成果。
- 取舍:不追求纯作品集的高度包装,也不做只给自己看的私人笔记库;换来一个既能长期生长、又能被外部读者快速理解的个人空间。
- 推翻条件:如果未来主要目标转向求职/接单转化,可提高作品展示权重;如果转向纯私人知识管理,可降低对外展示权重。

### D-20260613-公开边界
- 状态:已定
- 背景:站点目标是公开自留地,需要明确哪些内容可以进入发布流。
- 决策:公开边界默认**不限制内容类型**。学习笔记、半成品、项目过程、阶段计划、个人知识库输出、成熟长文等都可以发布。是否发布主要由后续 agent 整理/发布流程判断质量、隐私和可读性,而不是由内容类型预先限制。
- 取舍:获得更低发布摩擦和更真实的成长记录;代价是后续必须设计基本的发布前整理与安全检查,避免泄露 secrets、隐私或明显不可读内容。
- 推翻条件:如果后续出现隐私、职业风险或噪声过高的问题,再引入更严格的发布分级。

### D-20260613-内容源与发布流
- 状态:已定
- 背景:用户计划后期把个人知识库输出对接到 blog,并通过 agent 定时整理发布;因此内容源不应绑定到单一编辑器或旧站 Notion 流程。
- 决策:内容来源**不做单一限制**。长期方向是由 agent 从个人知识库、笔记、项目记录或其他来源整理成可发布内容,再进入 blog。Blog 侧关注发布后的内容模型与展示,不把某个工具(Notion/Obsidian/本地 Markdown)设为唯一源头。
- 取舍:获得最大输入弹性,适合 agent 驱动的长期知识发布;代价是后续需要定义一个稳定的“发布中间格式”和 agent 发布前检查流程。
- 推翻条件:如果多源输入导致维护成本过高或内容质量不稳定,再收敛到单一源头或主源 + 辅源模型。

### D-20260613-旧内容策略

- 背景:旧站内容来自过去实现,且用户明确表示旧内容可以完全废弃。
- 决策:旧站内容**可以完全废弃**,不作为新站迁移范围。后续如有个别旧内容仍有价值,只作为素材重写或重新整理,不做全量迁移。
- 取舍:牺牲旧内容连续性和历史归档;换来从最终目标出发的干净重建,避免旧结构污染新信息架构。
- 推翻条件:如果后续发现某些旧文章对作品展示或知识体系有明确价值,可作为单篇素材重新纳入,但不恢复全量迁移策略。

### D-20260613-首页叙事
- 状态:已定
- 背景:新站需要同时承担学习自留地和简历/作品展示入口,二者不应互斥。
- 决策:首页同时承载「我是谁 / 能力 / 作品」与「最近在学什么 / 做什么 / 正在形成什么判断」。不把首页单独压成纯简历页或纯动态流;后续信息架构需要让外部读者能快速理解 Shawn,也能进入持续生长的学习空间。
- 取舍:首页结构会比单一博客首页复杂;换来同时服务外部评价和长期自留地的目标。
- 推翻条件:如果后续首页过载或读者路径混乱,再拆成更明确的 landing page + garden/log 入口。

### D-20260613-一级内容类型
- 状态:已定
- 背景:需要先收敛最终支持的内容类型,避免实现时无限扩栏目。
- 决策:一级内容类型暂定为 `About`、`Projects`、`Notes`、`Logs/复盘`、`Essays`、`Links`。其中 `Notes` 承载知识卡片/学习笔记,`Logs/复盘` 承载阶段性过程记录,`Essays` 承载成熟长文,`Projects` 承载作品与构建记录,`Links` 承载友链/推荐/RSS 生态连接。
- 取舍:获得清晰的内容骨架;代价是未来新增栏目需要先证明不能被现有类型容纳。
- 推翻条件:实际发布一段时间后,如果某类内容明显无法归入现有类型,再新增一级类型。

### D-20260613-内容状态与同步方式
- 状态:已定
- 背景:内容主要会先在本地或个人知识库侧写好,输出到某个 output 后,再由 agent 自动同步到 blog;因此 blog 不需要承担复杂写作状态管理。
- 决策:blog 侧不优先设计复杂的 draft / seed / growing / evergreen 状态体系。后续需要在本项目提供一个同步功能或同步工具,供 Claude Code、Codex 等 agent 调用,把整理好的 output 内容发布到站点内容目录。
- 取舍:减少站点本身的状态复杂度;代价是同步工具需要承担格式转换、路径生成、元数据补齐和发布前检查。
- 推翻条件:如果公开内容越来越多且需要显式表达成熟度,再增加轻量状态字段。

### D-20260613-agent发布工具边界
- 状态:已定
- 背景:未来发布流会由不同 coding agent 调用,项目本身应提供工具/命令,而不是绑定某个具体 agent 产品。
- 决策:发布前检查与同步能力应作为本项目内的可调用工具提供,供 Claude Code、Codex 等 agent 使用。工具至少应覆盖 secrets/隐私检查、基本可读性/元数据检查、来源/摘要/标题补齐、输出路径规范化。agent 负责调用工具并处理结果。
- 取舍:获得跨 agent 的可复用发布入口;代价是需要维护工具约定和失败处理。
- 推翻条件:如果最终只使用单一 agent 且流程稳定,可把工具简化为少量脚本或命令。

### D-20260613-技术栈原则
- 状态:已定
- 背景:具体技术栈仍待选,但需要先定筛选原则,避免为了炫技选择过重方案。
- 决策:技术栈优先选择**静态站**方向,并且必须对 Markdown 友好、便于 agent 写入和同步、易生成 RSS、部署简单、尽量不依赖数据库或复杂后端。
- 取舍:牺牲动态交互和复杂后台能力;换来长期可维护、低成本、适合 agent 自动发布的架构。
- 推翻条件:如果后续明确需要登录、评论、会员、动态查询或复杂后台,再重新评估是否引入后端。

### D-20260613-上线标准
- 状态:已定
- 背景:上线不应按临时 MVP 降级,而应在核心目标完成后再切换。

- 取舍:上线会比极简首页更晚;换来切换时站点形态完整,不需要马上推倒重来。
- 推翻条件:如果需要先放一个临时 landing page 供外部使用,可另行定义临时上线标准,但不能替代最终上线标准。

### D-20260614-git与旧站处理策略
- 状态:已定
- 背景:当前仓库工作区仍包含旧 Hugo 站,但新站最终会替换旧站;用户明确表示不用围绕旧站保守规划,只需不要弄丢旧内容即可。
- 决策:后续按 Git flow / worktree 方式推进新站重建。旧站不作为实现目标,可以被新站替换;在删除或覆盖旧站文件前,必须确保旧站状态可通过 Git 历史、分支、tag 或 worktree 快照恢复。实现阶段优先在新分支或独立 worktree 中开展,保持每步小提交、可回退。
- 取舍:减少对旧站结构的迁就,提升重建速度;代价是切换前需要更清晰的分支/快照纪律,避免误删不可恢复内容。
- 推翻条件:如果后续决定保留旧站某部分实现或内容,需要重新定义迁移边界。

### D-20260614-部署目标
- 状态:已定
- 背景:新站是静态 Astro 站,现有线上地址已经在 github.io 体系中,用户希望完成后直接发到 GitHub。

- 取舍:部署链路简单、贴合现有 github.io;代价是预览协作能力弱于 Cloudflare Pages/Vercel preview。

### D-20260614-发布中间格式细节
- 状态:已定
- 背景:内容会由 agent 从本地 output 或知识库整理后同步进 blog,需要稳定但不繁重的中间格式。
- 决策:同步输入必须显式表达发布意图。初始支持单篇 Markdown;目录批量同步时使用显式 `manifest.json` 描述 `type`、`file`、`slug`、`publish` 等字段。manifest 字段后续由实现阶段按最小可用原则确定。
- 取舍:显式 manifest 降低 agent 误判类型和误发布风险;代价是批量发布前需要生成一个 manifest。
- 推翻条件:如果后续发现 manifest 增加太多摩擦,可增加目录自动推断模式,但默认仍保留显式 manifest。

### D-20260614-自动发布边界
- 状态:已定
- 背景:未来可能由 agent 定时任务整理并发布内容,但需要明确自动化边界。
- 决策:允许未来 agent 定时任务自动同步、提交或推送内容,前提是项目内发布前检查工具通过。当前实现阶段先提供可调用工具与本地检查;自动提交/推送可作为后续能力接入。
- 取舍:保留长期自动化能力;代价是必须认真实现 secrets/隐私/可读性/元数据检查,避免自动发布风险。
- 推翻条件:如果自动发布导致噪声、隐私或质量问题,改为生成 PR/草稿而不是直接推送。

### D-20260614-UI验证工具链
- 状态:已定
- 背景:agent 做 UI 的质量瓶颈在「能否看到自己渲染的结果再迭代」(见 `research.md`)。需要一套让 agent 在 ⑥自验证 里截图自检 + a11y 检查的工具,但要受「新增依赖先问」纪律约束,且不能给冻结期的部署链路添堵。
- 决策:新站验证工具链选 **Playwright(含 Chromium)+ axe-core**,作为 devDependency 进新站工具链。视觉自检经 **Playwright MCP** 截图回喂多模态模型;a11y 自检用 `@axe-core` 扫构建产物;性能/SEO 用一次性 `npx lighthouse` 跑基线、不固化为依赖。**三者全部只在本地 ⑥自验证 跑,不进 ⑧ 部署 CI(`withastro/action`)。** 回路细节见 `CONVENTIONS.md` 的「UI 自验证回路」。
- 取舍:获得 agent 视觉 + a11y 自闭环验证能力;代价是新站初始工具链多一个重量级依赖(Chromium 二进制数百 MB)与本地 install 步骤。换来不碰冻结期发布链路、与「无 preview 链路」决策(见 `D-20260614-部署目标`)一致。
- 推翻条件:若 Playwright 二进制 / 本地装机成本过高,退化为「仅 axe-core + 人工预览截图」;若后续需要 PR 级视觉 gate,再评估是否把验证接入 CI。

- 推翻条件:若脚本维护成本过高或固定回路需求消失,回退为纯人工预览 + MCP 探索;若需要 PR 级视觉 gate 再评估接入 CI。
